# Plan — Item Reference Generator

## Summary

Create a build-time Node.js script that parses all BMB item XML files and generates a comprehensive Markdown reference document at `docs/references/items.md`. The reference is categorised by item type (Weapons, Armor, Accessories, Consumables, Clothing) with a table of contents for quick navigation. The script integrates into the existing tooling as an `npm run reference` command and a new menu item.

## Architectural Context

The project uses Node.js ESM scripts under `scripts/` with zero npm dependencies. Existing scripts follow a consistent pattern:

- **`scripts/build.mjs`** — exports an async function (`build()`), uses `process.exit(1)` for errors, imports helpers from `scripts/lib/output.mjs`.
- **`scripts/menu.mjs`** — interactive menu using `execSync` to invoke scripts; automatically assigns alphabetical shortcut keys.
- **`scripts/lib/output.mjs`** — shared colour-aware console output helpers (`success`, `error`, `info`, `step`).
- **`package.json`** — defines `scripts.build`, `scripts.menu`, `scripts.prepare`. Has `fast-xml-parser` as a dependency (installed for this task).

All mod XML files live under `Mods/src/Data/GameCore/` and share a common structure: a `<GameItemTypes>` root containing `<GameItemType InternalName="...">` entries with child elements like `<DisplayName>`, `<Description>`, `<Type>`, `<ShopValue>`, `<RarityDisplay>`, `<GameModifier>`, etc.

The five item-containing XML source files are:

| File | Content | ~Count |
|---|---|---|
| `Mods/src/Data/GameCore/BMB_Items.xml` | Accessories + consumables | 86 |
| `Mods/src/Data/GameCore/BMB_Weapons.xml` | Weapons | 71 |
| `Mods/src/Data/GameCore/BMB_Armor.xml` | Armor (head, torso, forearms, legs, boots, shields, cloaks) | 85 |
| `Mods/src/Data/GameCore/BMB_Clothes.xml` | Clothing (hair, shirts, surcoats, pants) | 19 |
| `Mods/src/Data/GameCore/BMB_CoreItemsModifications.xml` | Base game overrides (1 item) | 1 |

## Approach / Architecture

### New files

1. **`scripts/generate-reference.mjs`** — The generator script. Reads the XML files, parses item data, organises by category, and writes `docs/references/items.md`.
2. **`docs/references/items.md`** — The generated output (git-tracked so it's always available without running the script; regenerated on demand).

### XML Parsing Strategy

Use **`fast-xml-parser`** (already installed as a dependency) for robust XML parsing. This library is pure JavaScript with zero transitive dependencies, making it a lightweight addition. Using a proper parser over regex is preferred because:

- It handles edge cases (CDATA, entities, attribute quoting variations) without custom logic.
- The project will likely need more XML processing in the future, so the dependency pays for itself.
- It supports `isArray` configuration to correctly handle tags that may appear once or multiple times (e.g., `<Type>`, `<GameModifier>`).

Configuration notes for `fast-xml-parser`:
- `ignoreAttributes: false` — needed to read `InternalName` from `<GameItemType InternalName="...">` attributes.
- `attributeNamePrefix: '@_'` — keeps attributes distinct from child elements.
- `isArray` callback — must return `true` for tags that can repeat: `Type`, `GameModifier`, `Prereq`, `SupportedUnitModelType`, etc.

### Category Organisation

Items are grouped into sections based on their source file and `<Type>` values:

| Section | Source | Filter |
|---|---|---|
| **Weapons** | `BMB_Weapons.xml` | All items |
| **Armor — Head** | `BMB_Armor.xml` | `<Type>Head</Type>` |
| **Armor — Torso** | `BMB_Armor.xml` | `<Type>Torso</Type>` |
| **Armor — Forearms** | `BMB_Armor.xml` | `<Type>Forearms</Type>` |
| **Armor — Legs** | `BMB_Armor.xml` | `<Type>LowerBody</Type>` |
| **Armor — Boots** | `BMB_Armor.xml` | `<Type>Boots</Type>` |
| **Armor — Shields & Defense** | `BMB_Armor.xml` | `<Type>Defense</Type>` |
| **Armor — Cloaks & Surcoats** | `BMB_Armor.xml` | `<Type>Surcoat</Type>` |
| **Accessories** | `BMB_Items.xml` | Items with `<CanBeEquipped>1</CanBeEquipped>` |
| **Consumables** | `BMB_Items.xml` | Items with `<IsUsable>1</IsUsable>` and no `CanBeEquipped` |
| **Clothing** | `BMB_Clothes.xml` | All items |

`BMB_CoreItemsModifications.xml` is excluded — it modifies base game items, not new BMB content.

### Markdown Output Structure

```markdown

> Auto-generated by `npm run reference`. Do not edit manually.

**242 items** across 5 categories.

## Table of Contents

- [Weapons (71)](#weapons)
- [Armor (85)](#armor)

  - [Torso (N)](#armor--torso)
  - ...
- [Accessories (N)](#accessories)
- [Consumables (N)](#consumables)
- [Clothing (19)](#clothing)

## Weapons

| Name | Rarity | Value | Effects |
|---|---|---|---|
| Fire and Ice | Uncommon | 160 | +12 Pierce Attack, +3 Fire Attack, ... |
| ... | ... | ... | ... |

### Head

| Name | Rarity | Value | Effects |
|---|---|---|---|
| ... | ... | ... | ... |

(etc.)
```

### Effects Summary

The `<GameModifier>` elements are translated into a human-readable effects string:

- `AdjustUnitStat` + `StrVal` + `Value` → "+{Value} {stat}" (e.g., "+2 Poison Attack")
- `Provides` text if present → appended as-is (it's already human-readable).
- `UnlockCombatAbility` / `MeleeDefenseAppliesSpell` / etc. → use the `Provides` text.

Stat names are cleaned up (strip `UnitStat_` prefix, insert spaces before capitals).

### Integration

- **`package.json`**: Add `"reference": "node scripts/generate-reference.mjs"` to the scripts block.
- **`scripts/menu.mjs`**: Add a new menu item `"Generate item reference (docs/references/)"` that runs the script.
- **`README.md`**: Add the new menu item to the "Available menu items" table.
- **`AGENTS.md`**: Add `docs/references/items.md` to the File Map and note the `npm run reference` script.

## Rationale

- **Library over regex**: `fast-xml-parser` is a zero-transitive-dependency, pure-JS library. Using a proper parser is more maintainable than regex and positions the project for future XML processing tasks.
- **Git-tracked output**: The generated `items.md` is committed so it's always browsable on GitHub / in the repo without running the script. A header comment marks it as auto-generated.
- **Source file as category**: Using the source XML file as the primary grouping (Weapons, Armor, Items, Clothes) aligns with how the mod author already organises content, making the reference intuitive.
- **Armor sub-sections**: Armor has 85 items across 7+ slot types; a single flat table would be unwieldy. Sub-sections by slot type improve navigability.

## Detailed Steps

1. **Create `docs/references/` directory** — will be created automatically when the script writes `items.md`.

2. **Create `scripts/generate-reference.mjs`**:
   - Import `node:fs/promises` (`readFile`, `writeFile`, `mkdir`), `node:path`, and `scripts/lib/output.mjs`.
   - Define constants for the source XML file paths (relative to `PROJECT_ROOT`).
   - Implement `parseItems(xmlContent)` — splits XML into `<GameItemType>` blocks, extracts fields per block, returns an array of item objects.
   - Implement `formatEffects(modifiers)` — converts `GameModifier` data into a readable effects string.
   - Implement `formatStatName(raw)` — cleans `UnitStat_Attack_Poison` → `Poison Attack`.
   - Implement `categorise(items, sourceFile)` — assigns items to their display category based on source file and `<Type>`.
   - Implement `generateMarkdown(categories)` — builds the full Markdown string with TOC, section headers, and tables.
   - Implement and export `generateReference()` — orchestrator: reads files, parses, categorises, generates, writes.
   - Add direct-invocation guard (same pattern as `build.mjs`).

3. **Add `npm run reference`** — add `"reference": "node scripts/generate-reference.mjs"` to `package.json` scripts.

4. **Add menu item** — insert a new entry in `scripts/menu.mjs` `menuItems` array: `{ label: "Generate item reference (docs/references/)", action: () => runCommand("node scripts/generate-reference.mjs") }`.

5. **Update `README.md`** — add the new menu item key (`c`) to the "Available menu items" table.

6. **Update `AGENTS.md`** — add `docs/references/items.md` to the File Map; add `npm run reference` entry to the Project Stats section.

7. **Run the script** — execute `npm run reference` to generate the initial `docs/references/items.md` and verify output.

## Assumptions

- The XML file structure will remain stable (same element names, same nesting patterns).
- `<GameItemType>` blocks do not span across XML files (each is self-contained within one file).
- All items worth documenting have a `<DisplayName>` element.
- `BMB_CoreItemsModifications.xml` should be excluded from the reference since it modifies base game items rather than adding new ones.
- The `<Provides>` text inside `<GameModifier>` is already human-readable and can be used directly in the effects column.

## Constraints

- **OS-independence**: All paths via `node:path`, all I/O via `node:fs/promises` (per `AGENTS.md` §6).
- **UTF-8**: Read XML as UTF-8, write Markdown as UTF-8 (per encoding rules).
- **No external dependencies**: Regex-based XML extraction, no npm packages.
- **Existing patterns**: Follow the same script structure as `build.mjs` (exported async function, direct-invocation guard, shared output helpers).

## Out of Scope

- Spells reference (`BMB_Spells.xml`) — different schema, different use case; can be added later.
- Abilities reference (`BMB_Abilities.xml`) — same reasoning.
- Units reference (`BMB_Units.xml`, `BMB_UnitStats.xml`) — different schema.
- Automated regeneration on build (`npm run build` does not trigger reference generation).
- CI/CD integration or pre-commit hooks.
- Interactive filtering or search in the generated Markdown.

## Acceptance Criteria

- Running `npm run reference` generates `docs/references/items.md` without errors.
- The generated file contains a table of contents with working anchor links.
- Items are grouped by category: Weapons, Armor (by slot), Accessories, Consumables, Clothing.
- Each item row shows: Name, Rarity, Shop Value, and a human-readable Effects summary.
- Items are sorted alphabetically by DisplayName within each section.
- The script follows existing project conventions (ESM, shared output helpers, OS-independent paths).
- The menu item appears as option `c` in `npm run menu`.
- The generated file includes a "do not edit manually" header.
- Section headers include item counts matching the actual parsed data.

## Testing Strategy

- **Script execution**: Run `npm run reference` and verify it exits with code 0.
- **Output validation**: Open `docs/references/items.md` and verify:
  - TOC links navigate to correct sections.
  - All 4 source files are represented (weapons, armor, items, clothes).
  - Item counts per section are plausible (e.g., ~71 weapons, ~85 armor, etc.).
  - Effects column contains readable stat summaries, not raw XML.
  - No items have empty Name or missing data.
- **Menu integration**: Run `npm run menu`, press `c`, verify the reference script runs.
- **Idempotency**: Run the script twice — output should be identical.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| **XML parser misconfigures array tags** | Configure `isArray` for all repeatable tags (`Type`, `GameModifier`, `Prereq`, etc.). Test against all 4 files. If an item fails to parse, log a warning and skip it rather than crashing. |
| **New XML files added later** | The source file list is a constant array in the script; adding a new file requires a one-line addition. Document this in a code comment. |
| **GameModifier schema varies** | Extract only well-known attributes; fall back to `Provides` text or "—" for unrecognised modifier types. |
| **Item count changes** | Counts in the TOC are dynamically calculated from parsed data, never hardcoded. |
| **DisplayName missing** | Fall back to InternalName if DisplayName is absent. |
