# Plan: Reference Generators for All Data Types

## Summary

Create dedicated reference generator scripts for the five non-item data types added by the Black Market Bazaar mod: **spells**, **abilities**, **units**, **unit stats**, and **effects**. Each generator follows the same architectural pattern as the existing item reference generator (`scripts/generate-reference.mjs`) — reading from the monolithic GameCore XML files, resolving `TXT_BMB_*` localization keys, and producing a Markdown reference document in `docs/references/`. A new umbrella npm script and menu entry will allow generating all references at once.

## Architectural Context

### Existing reference pipeline

The item reference generator at [`scripts/generate-reference.mjs`](scripts/generate-reference.mjs) establishes the pattern:

1. Calls [`mergeXmlFragments()`](scripts/lib/merge-xml.mjs:164) to ensure monolithic XML files are up-to-date.
2. Calls [`mergeTranslations()`](scripts/lib/merge-translations.mjs) to ensure localization files are current.
3. Loads English localization keys via [`parseLocalizationXml()`](scripts/lib/xml-parser.mjs:154) from `Mods/src/Data/Localization/English/`.
4. Parses GameCore XML via [`parseGameCoreXml()`](scripts/lib/xml-parser.mjs:199).
5. Categorises, formats, and writes Markdown to [`docs/references/items.md`](docs/references/items.md).

### Source XML files (generated from fragments)

| Data Type | Monolithic File | Root Element | Entry Element | Fragment Count |
|---|---|---|---|---|
| Spells | `BMB_Spells.xml` | `<Spells>` | `<SpellDef>` | ~81 |
| Abilities | `BMB_Abilities.xml` | `<AbilityBonuses>` | `<AbilityBonus>` → `<AbilityBonusOption>` | 2 |
| Units | `BMB_Units.xml` | `<UnitTypes>` | `<UnitType>` | 22 |
| Unit Stats | `BMB_UnitStats.xml` | `<PlayerAbilityTypes>` | `<UnitStatType>` | 3 |
| Effects | `BMB_Effects.xml` | `<EffectBlueprints>` | `<EffectSequencer>` | 32 |

### Shared modules available

- [`scripts/lib/xml-parser.mjs`](scripts/lib/xml-parser.mjs) — `parseGameCoreXml()`, `parseLocalizationXml()`, `escapeXml()`
- [`scripts/lib/merge-xml.mjs`](scripts/lib/merge-xml.mjs) — `mergeXmlFragments()`
- [`scripts/lib/merge-translations.mjs`](scripts/lib/merge-translations.mjs) — `mergeTranslations()`
- [`scripts/lib/output.mjs`](scripts/lib/output.mjs) — `success()`, `error()`, `warn()`, `info()`, `step()`

### Key conventions

- All reference docs are auto-generated; header includes `> Auto-generated by \`npm run reference\`. Do not edit manually.`
- Items are sorted alphabetically within categories.
- Localization keys (`TXT_BMB_*`) are resolved to English text before display.
- The `GAMECORE_ARRAY_TAGS` set in [`xml-parser.mjs`](scripts/lib/xml-parser.mjs:179) may need extension for new entry element types (e.g., `SpellDef`, `UnitType`, `UnitStatType`, `AbilityBonus`, `AbilityBonusOption`, `EffectSequencer`, `EmitterBlueprint`, `SequenceFrame`).

| `scripts/generate-reference-effects.mjs` | `docs/references/effects.md` | `BMB_Effects.xml` |

### Shared infrastructure extraction

To avoid duplicating the merge-then-load-localization boilerplate across 6 scripts (the existing item generator + 5 new ones), extract common logic into a new shared module:

| Module (NEW) | Path | Purpose |
|---|---|---|
| `reference-helpers.mjs` | `scripts/lib/reference-helpers.mjs` | Shared functions: `ensureFreshBuild()` (merge XML + translations), `loadLocalizationKeys()`, `resolveKey()`, `escapeCell()`, `formatInternalName()` |

Then refactor [`scripts/generate-reference.mjs`](scripts/generate-reference.mjs) to import from this shared module instead of defining these functions inline.

### Umbrella script

Modify the existing [`scripts/generate-reference.mjs`](scripts/generate-reference.mjs) to become an orchestrator that:
1. Runs the merge steps once (not per-generator).
2. Calls each individual generator's exported function.
3. Reports a combined summary.

Alternatively, create a new `scripts/generate-all-references.mjs` that imports and calls all generators. The existing `generate-reference.mjs` remains focused on items only.

**Recommended approach:** Create `scripts/generate-all-references.mjs` as the umbrella. Each individual generator exports a function that accepts a pre-loaded `keyMap` (localization keys) to avoid redundant merge/load cycles. The umbrella script runs merge once, loads keys once, then passes them to each generator.

## Rationale

- **Separate files per data type** keeps each generator focused and maintainable. Spells have very different fields from units or effects.
- **Shared helper module** eliminates code duplication for merge/localization/formatting logic that's identical across all generators.
- **Umbrella script** provides a single command to regenerate everything, while individual scripts allow targeted regeneration during development.
- **Effects get a simpler generator** since they have no localization — just a listing of particle effect names and their emitter counts.
- **`parseGameCoreXml()` reuse** — the existing parser handles all GameCore XML formats. The `isArray` configuration may need extension for new repeating tags in spell/unit/ability XML.

## Detailed Steps

### Step 1: Extend `xml-parser.mjs` array tag support

Add the following tags to `GAMECORE_ARRAY_TAGS` in [`scripts/lib/xml-parser.mjs`](scripts/lib/xml-parser.mjs:179):

- `SpellDef` — multiple spells in `<Spells>` root
- `UnitType` — multiple units in `<UnitTypes>` root
- `UnitStatType` — multiple stats in `<PlayerAbilityTypes>` root
- `AbilityBonus` — multiple abilities in `<AbilityBonuses>` root
- `AbilityBonusOption` — multiple options within `<AbilityBonus>`
- `EffectSequencer` — multiple effects in `<EffectBlueprints>` root
- `EmitterBlueprint` — multiple emitters within `<EffectSequencer>`
- `SequenceFrame` — multiple frames within `<EffectSequencer>`
- `SelectedAbilityBonusOption` — multiple selections in `<UnitType>`
- `Equipment` — multiple equipment entries in `<UnitType>`
- `EquipmentUpgradeDef` — multiple upgrade defs in `<UnitType>`
- `LevelMilestone` — multiple milestones in `<UnitType>`
- `SpellDefEffect` — multiple effects in `<SpellDef>`
- `HitSoundFX` — multiple sounds in `<SpellDef>`
- `PreventStackingWith` — multiple stacking rules in `<SpellDef>`
- `SpellResourceCost` — multiple resource costs in `<SpellDef>`
- `AIData` — AI data blocks
- `Calculate` — multiple calculate blocks in `<GameModifier>`

### Step 2: Create shared reference helpers module

**New file:** `scripts/lib/reference-helpers.mjs`

Extract from [`scripts/generate-reference.mjs`](scripts/generate-reference.mjs):
- `ensureFreshBuild()` — calls `mergeXmlFragments()` + `mergeTranslations()`, returns summary info
- `loadLocalizationKeys()` — loads all English localization XML files, returns `Map<key, text>`
- `resolveKey(value, keyMap)` — resolves `TXT_BMB_*` keys to English text
- `escapeCell(text)` — escapes pipe characters and newlines for Markdown table cells
- `formatInternalName(raw)` — converts `BMB_SomeName` to `Some Name`
- `formatStatName(raw)` — converts `UnitStat_Attack_Fire` to `Fire Attack` (with `STAT_LABELS` map)
- Constants: `PROJECT_ROOT`, `GAMECORE_DIR`, `LOCALIZATION_EN_DIR`

### Step 3: Refactor existing item reference generator

Update [`scripts/generate-reference.mjs`](scripts/generate-reference.mjs) to import shared functions from `scripts/lib/reference-helpers.mjs` instead of defining them inline. The item-specific logic (parsing `GameItemType`, categorisation, effect formatting) stays in the file. The exported `generateReference()` function signature changes to optionally accept a pre-loaded `keyMap`:

```js
export async function generateReference(keyMap = null)
```

If `keyMap` is `null`, the function runs merge + loads keys itself (standalone mode). If provided, it skips the merge/load step (umbrella mode).

### Step 4: Create spells reference generator

**New file:** `scripts/generate-reference-spells.mjs`  
**Output:** `docs/references/spells.md`

Reads `BMB_Spells.xml`, parses `<SpellDef>` entries. Reference table columns:

| Column | Source |
|---|---|
| Name | `DisplayName` (resolved from `TXT_BMB_*`) |
| Type | `SpellType` (Tactical/Strategic) |
| Class | `SpellClass` (Offensive/Defensive) |
| Target | `SpellTargetType` |
| Cost | `StaminaCost` or `SpellResourceCost` |
| Description | `Description` (resolved from `TXT_BMB_*`) |

Categorise spells by `SpellType` (Tactical vs Strategic) or by `SpellClass` (Offensive/Defensive/Buff). Sort alphabetically within categories.

## Assumptions

- The monolithic XML files (`BMB_Spells.xml`, `BMB_Units.xml`, etc.) are always generated fresh before reference generation (the generators call merge first).
- The `parseGameCoreXml()` parser can handle all GameCore XML formats once the array tags are extended.
- Effects don't need localization key resolution (their `DisplayName` values are plain text, not `TXT_BMB_*` keys).
- The existing `formatEffects()` logic from the item generator can be reused (with minor adaptations) for spell and ability modifiers.

## Constraints

- All scripts must use ESM modules (`import`/`export`) consistent with the project's `"type": "module"` setting in `package.json`.
- All generated Markdown files must include the `> Auto-generated` header to prevent manual editing.
- File encoding must be UTF-8 (no BOM) per project constraints.
- Scripts must be OS-independent (no Windows-specific paths or commands).

## Out of Scope

- **`core-items-mods/` reference** — only 1 base game override exists; not worth a dedicated reference doc.
- **Cross-referencing between data types** — e.g., linking spell names in item effects to the spells reference. This would be a future enhancement.
- **Filtering or search functionality** — the references are static Markdown files.
- **Non-English reference generation** — references are always in English.
- **Automated testing** — the project uses manual in-game smoke testing; no unit test framework exists.

## Acceptance Criteria

1. Running `npm run reference:all` generates all 6 reference documents (items + 5 new) in `docs/references/`.
2. Each reference document has a header, item count, table of contents (where applicable), and formatted tables.
3. All `TXT_BMB_*` keys in translatable data types are resolved to English text.
4. Running `npm run reference` still generates only the items reference (backward compatibility).
5. Each individual generator (`npm run reference:spells`, etc.) works standalone.
6. The interactive menu `[c]` option generates all references.
7. All generated reference files include the `> Auto-generated` warning header.
8. No existing functionality is broken by the refactoring.

## Testing Strategy

1. **Build verification:** Run `npm run build` to ensure XML merge still works after `xml-parser.mjs` changes.
2. **Individual generator testing:** Run each `npm run reference:<type>` script and verify the output Markdown is well-formed and contains expected entries.
3. **Umbrella testing:** Run `npm run reference:all` and verify all 6 files are generated.
4. **Backward compatibility:** Run `npm run reference` and verify `items.md` output is identical to the pre-refactor version.
5. **Menu testing:** Run `npm run menu`, select `[c]`, and verify all references are generated.
6. **Spot-check content:** Verify a few entries in each reference doc against the source XML fragments to ensure correct data extraction and localization resolution.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| **`parseGameCoreXml()` may not handle all XML structures correctly after array tag extension** | Test with actual generated XML files; the parser is already battle-tested with item XML which has similar complexity |
| **Adding too many tags to `GAMECORE_ARRAY_TAGS` could break existing item parsing** | Only add tags that actually repeat in the new data types; verify items.md output is unchanged after the change |
| **Spell XML has complex nested structures (`Calculate`, `AIData`, `ValueCalcWrapper`)** | The reference generator only needs surface-level fields (name, type, cost, description); deep nesting can be ignored |
| **Effects XML uses `EffectSequencer` not `EffectBlueprint` as the entry element** | Verified from actual fragment files — the root is `<EffectBlueprints>` but entries are `<EffectSequencer>` |
| **Refactoring `generate-reference.mjs` could introduce regressions** | Compare items.md output before and after refactoring to ensure byte-identical results |
